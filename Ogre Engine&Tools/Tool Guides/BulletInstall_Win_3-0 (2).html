Cloning the Bullet Source

Using the Linux commandline or a suitable client on Windows (e.g. gitbash) clone the bullet repository to your local machine.

> git clone https://github.com/bulletphysics/bullet3.git

Versions

Due to the nature of the Bullet project (open source, active, being developed,) it's difficult to keep this guide and the examples in sync with the current version of Bulet available on GitHub. These notes are written against 3.2.5 please checkout that tag before continuing!

> git checkout tags/3.25

Using CMake to Generate a Compilation Environment

Make sure CMake is pointing at the top level of the bullet3 repository (where the top level CMakeLists file is,) you'll need to set the location of the build files to a build directory within this one. So if your path is C:\MyStuff\bullet3 the path for your build files will be C:\MyStuff\bullet3\build, some of the steps which follow work on the basis that you have this layout - please don't deviate.

Open the Cmake Gui and set the parameters as shown in the tables below, you may have to amend the paths to match your repo location. Note the CMAKE_INSTALL_PREFIX this is where we are putting our compiled libraries and the header files which make up the Bullet library. I've changed this to make this more consistent with Ogre3D, in that I'm writing the output to an sdk directory.

CMake Build Entries

Please note, you won't need to set all of these. I'm pretty sure it's just the ones which are in bold, please make to set:

  • CMAKE_INSTALL_PREFIX

These should set automatically ...

  • LIBRARY_OUTPUT_PATH
  • INCLUDE_INSTALL_DIRECTORY

Are set correctly, the path should end inside the clone of bullet in the build directory.

You'll also need to set the flags for:

  • INSTALL_CMAKE_FILES
  • INSTALL_LIBS
  • INSTALL_EXTRA_LIBS
  • USE_MSVC_RUNTIME_LIBRARY_DLL

Building and Installing

We build the project in the same way we would for any other project in our build environment. One notable difference is we must build the install target, this copies the files to another location from which they can run. This will be familiar to those of you with Mac, Linux or Unix machines at home who have installed software from source. There is a nice explanation of this here anyone would like to know more.

Now its simply a question of opening visual studio building each of the configurations:

  • Debug (Not required this year?)
  • Release
  • Release_withdebuginfo

We'll be linking our project against one or other of these (probably RelWithDebInfo), essentially they have to match i.e. if your compiling a Release version of your game (which uses Bullet) you'll need the Release version. The issue with this is that the Debug version has a lot of extras we probably don't need, so release with debug information provides us with a release build with some of the debugging information (a sort of halfway house).

After each build we build the install target, which copies them to a directory. If you followed the instructions this is the sdk directory within the build directory. This is a convention I've adopted from Ogre for consistency.

NOTE: You MUST compile and install both Release and RelWithDebInfo while we'll probably only use the latter the cmake script which looks for the installed version of Bullet looks for the release library.
